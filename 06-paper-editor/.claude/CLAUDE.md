# 06-Paper-Editor — 기존 논문 교정 및 모의 피어리뷰 하네스

사용자가 제공한 기존 논문(.tex, .pdf, .md)을 입력으로 받아,
**문법 교정 / 팩트체크 / 모의 피어리뷰 / 구조 개선 제안** 등
선택적 편집 서비스를 제공하는 AI 하네스.

> **범위**: 이 하네스는 기존 논문의 품질 개선(교정/리뷰)을 담당한다. 연구 결과로부터 새 논문을 작성하는 것은 범위 밖이다.

## 에이전트 팀 구성표

| 에이전트 | 역할 | 투입 모드 |
|---|---|---|
| **grammar-editor** | 문법·표현·일관성 교정 | 문법 교정 |
| **fact-checker** | 수치·주장을 코드/데이터/Figure와 대조 검증 | 팩트체크 |
| **structure-advisor** | 섹션 구성, 논리 흐름, Figure/Table 배치 분석 및 개선안 제시 | 구조 개선 |
| **reviewer** | 모의 리뷰어. 한 줄씩 팩트체크 + 방법론/데이터/해석 평가. 1회 리뷰 후 사용자 전달 | 모의 리뷰 |
| **reference-finder** | NASA ADS / arXiv에서 참고문헌 검색·추천. 출처 없는 주장에 레퍼런스 제안 | 팩트체크, 모의 리뷰 |

## 실행 모드 분기

사용자가 기존 논문 파일을 제공하면, **AskUserQuestion**으로 작업 모드를 제시한다.

| 모드 | 설명 | 실행 에이전트 | 아키텍처 패턴 |
|---|---|---|---|
| **문법·표현 교정** | 영문 문법, 학술 표현, 용어 일관성, 오타 교정. 과학적 내용 변경 안함 | grammar-editor | 단일 에이전트 |
| **팩트체크** | 논문의 **모든 문장**을 한 문장씩 읽으며 과학적 사실 정확성 검증. 출처 없는 주장에는 reference-finder가 레퍼런스 추천 | fact-checker + reference-finder | 순차 (fact-checker → reference-finder) |
| **모의 리뷰** | 리뷰어 1명이 한 줄씩 팩트체크 + 방법론/해석 평가. 1회 리뷰 후 리포트를 사용자에게 전달 | reviewer | 단일 에이전트 |
| **구조 개선 제안** | 섹션 구성, 논리 흐름, Figure/Table 배치 개선안 제시. 원고 자체는 수정하지 않음 | structure-advisor | 단일 에이전트 |

### 분기 규칙

1. 사용자가 `.tex`, `.pdf`, `.md` 파일 경로를 명시적으로 제공했을 때 이 분기를 활성화한다.
2. 사용자가 요청에서 작업 유형을 이미 특정한 경우 (예: "리뷰해줘", "문법 수정해줘") 메뉴를 건너뛰고 해당 모드로 바로 진입한다.
3. 복수 선택 가능: 예를 들어 "문법 교정 + 모의 피어리뷰"를 동시에 선택하면 grammar-editor 교정 후 mock-editor 리뷰 루프로 이어진다.
4. 모든 모드의 산출물은 `{작업경로}/` 아래에 저장한다.

### 모드 선택 질문 (AskUserQuestion)

```
질문: "제공하신 논문에 대해 어떤 작업을 수행할까요?"
헤더: "작업 선택"
복수선택: true
옵션:
  - 문법·표현 교정: 영문 문법, 학술 표현, 용어 일관성, 오타를 한 문장씩 교정합니다.
  - 팩트체크: 논문의 수치·주장을 제공된 코드/데이터/Figure와 대조 검증합니다.
  - 모의 리뷰: 리뷰어가 한 줄씩 팩트체크하고 방법론/해석을 평가합니다 (1회 리뷰).
  - 구조 개선 제안: 섹션 구성, 논리 흐름, Figure/Table 배치 개선안을 제시합니다.
```

## 작업 경로 정책

- 파이프라인 시작 시 사용자에게 **작업 경로를 질문**한다. 사용자가 쿼리에 경로를 명시했으면 그대로 사용한다.
- 입력 논문과 동일 디렉토리를 작업경로로 사용해도 무방하다.
- 이후 모든 `_workspace/` 참조는 사용자가 지정한 `{작업경로}`로 치환된다.

## 데이터 전달 규칙

| 에이전트 | 출력 파일 |
|---|---|
| grammar-editor | `{작업경로}/grammar/edited_draft.md`, `{작업경로}/grammar/grammar_report.md` |
| fact-checker | `{작업경로}/factcheck/factcheck_report.md` |
| structure-advisor | `{작업경로}/structure/structure_report.md` |
| reviewer | `{작업경로}/review/review_report.md` |
| reference-finder | `{작업경로}/references/reference_report.md` |
| 공통 | `{작업경로}/editorial-log.md` (전 과정 진행 기록 누적) |

**전달 규칙:**
1. 각 에이전트는 자신의 지정 파일에만 쓴다
2. 다른 에이전트의 출력은 읽기 전용으로 참조한다
3. 모든 중간 산출물은 삭제하지 않고 보존한다
4. `{작업경로}/editorial-log.md`에 모든 에이전트가 판단 과정을 누적 기록한다

## 입력 소스

- 사용자 직접 제공 논문 파일 (.tex, .pdf, .md)
- (선택) 팩트체크용 보조 자료: 코드, 데이터, Figure 이미지
- 자연어 요청 ("이 논문 문법 수정해줘", "리뷰해줘", "팩트체크해줘")

## 모의 리뷰 프로세스 상세

```
논문 입력
    │
    ▼
[reviewer] 한 줄씩 팩트체크 + 방법론/데이터/해석 평가
    │
    ▼
리뷰 리포트 (Strengths / Major / Minor / Fact-checks / Recommendation)
    │
    └─── 사용자에게 전달
```

> **참고**: 06-paper-editor는 리비전 루프를 자동으로 돌지 않는다. 1회 리뷰 결과를 사용자에게 전달하며, 사용자가 수정 후 재리뷰를 요청하면 같은 프로세스를 다시 실행한다.

## 기술 스택

| 범주 | 도구 |
|---|---|
| 언어 | Python 3.10+ |
| 논문 읽기 | pypdf, pdfplumber (PDF), Read 도구 (.tex, .md) |
| 텍스트 분석 | Grep, 정규식 기반 용어 스캔 |

## 사용 언어

- 사용자 대면: 한국어
- 교정 결과/리포트: 논문 언어에 맞춤 (영어 논문이면 영어 리포트)
- 리뷰 리포트: 영어

## 핵심 원칙

1. **원문 존중**: 저자의 의도와 문체를 존중하며, 불필요한 리라이팅을 하지 않는다
2. **근거 기반 지적**: 모든 지적에는 원문 인용 + 구체적 이유 + 수정안을 포함한다
3. **모드 분리**: 문법 교정은 내용을 바꾸지 않고, 팩트체크는 문법을 건드리지 않는다
4. **비파괴적 작업**: 원본 파일을 수정하지 않는다. 교정본/리포트는 별도 파일로 저장한다
5. **사용자 승인 필수**: 작업 모드와 대상 저널을 확인하고 승인 후 진행한다
6. **1회 리뷰 후 전달**: 모의 리뷰는 1회 수행 후 결과를 사용자에게 전달한다. 자동 리비전 루프 없음
7. **투명성**: 교정하지 않은 항목과 그 이유도 명시한다
8. **출처 기반 팩트체크**: 팩트체크는 모든 문장을 한 문장씩 검증하며, 과학적 주장에는 출처가 있어야 한다. 출처 없는 주장은 reference-finder가 ADS/arXiv에서 적절한 레퍼런스를 추천한다
