# Research-Workflow — 아이디어→실험→논문 통합 워크플로우

## 개요

연구자의 요청을 세 가지 경로로 처리한다:
1. **연구 업무 수행**: 자연어 작업 요청 → 계획 → 실행 → 결과 보고
2. **아이디어→실험**: 아이디어 → 실험 설계 → 피드백 → 실행 → 결과 해석
3. **논문 작성**: 실험 결과 → 논문 초안 작성

---

## 경로 1: 연구 업무 수행

사용자의 자연어 작업 요청(데이터 처리, 분석, 시뮬레이션 등)을 자동 처리한다.

### 에이전트 구성표

| 에이전트 | 역할 | Phase |
|---|---|---|
| **research-planner** | 요청 분석, 실행 계획 수립 | Phase 1 |
| **data-engineer** | 데이터 수집 | Phase 3 내부 |
| **research-executor** | 계획에 따른 실행 | Phase 3 |
| **paper-writer** | 결과 정리 및 보고 | Phase 4 |

### 워크플로우

```
[사용자 요청]
    │
    ▼
╔═══════════════════╗
║     Phase 1       ║
║  research-planner ║
║ (요청 분석/계획)    ║
╚════════╤══════════╝
         │ 실행 계획 (Markdown + JSON)
         ▼
╔═══════════════════╗
║     Phase 2       ║
║  사용자 승인/수정   ║  ◀─── 사용자 피드백 (최대 3회)
╚════════╤══════════╝
         │ 승인됨
         ▼
╔═══════════════════╗
║     Phase 3       ║
║ research-executor ║
║ ├ data-engineer   ║
║ ├ 전처리          ║
║ └ 모델 실행       ║
╚════════╤══════════╝
         │ 실행 결과
         ▼
╔═══════════════════╗
║     Phase 4       ║
║   paper-writer    ║
╚════════╤══════════╝
         │ 결과 보고
         ▼
╔═══════════════════╗
║     Phase 5       ║
║   피드백 루프      ║  ◀─── "이걸로 PFSS 돌려줘"
║ (후속 작업 요청)    ║ ────▶ Phase 1로 복귀
╚═══════════════════╝
```

### Phase별 상세

#### Phase 1: 요청 분석 및 계획 수립

1. 사용자의 자연어 요청을 research-planner에 전달한다.
2. 이전 작업의 컨텍스트(있는 경우)도 함께 전달한다.
3. research-planner가 실행 계획을 반환한다.

#### Phase 2: 사용자 승인

1. 계획을 사용자에게 제시한다.
2. 사용자 응답을 대기한다:
   - **승인**: "좋아", "진행해" → Phase 3
   - **수정**: "PHI 대신 HMI만으로 해줘" → research-planner 재호출 (최대 3회)
   - **거부**: "취소" → 작업 종료

#### Phase 3: 작업 실행

1. 승인된 계획을 research-executor에 전달한다.
2. 단계별 실행, 검증, 에러 대응.
3. 장시간 단계는 사용자에게 중간 상태를 알린다.

#### Phase 4: 결과 보고

1. 결과를 paper-writer가 보고서로 정리한다.
2. 성공/실패/부분성공 모두 보고한다.
3. 후속 작업 가능성을 제안한다.

#### Phase 5: 피드백 루프

1. 사용자가 후속 작업을 요청하면 Phase 1로 복귀한다.
2. 이전 작업의 출력을 context에 포함시킨다.

## 경로 2: 아이디어→실험

연구 아이디어를 실험 결과로 전환한다.

### 에이전트 구성표

| 에이전트 | 역할 | Phase |
|---|---|---|
| **research-planner** | 아이디어 분석, 실험 설계, 결과 해석 | Phase 1, 5 |
| **reviewer** | 실험 설계 검토 (옵션) | Phase 1 후 |
| **data-engineer** | 실험용 데이터 수집 | Phase 4 내부 |
| **research-executor** | 실험 파이프라인 실행 | Phase 4 |
| **paper-writer** | 실험 결과 보고 | Phase 5 |

### 워크플로우

```
[연구자: 아이디어 제시]
    │
    ▼
╔═════════════════════════╗
║   Phase 1: 아이디어 분석 ║
║   research-planner      ║
║   (질문 파싱, 가설 수립)  ║
╚════════╤════════════════╝
         │ 실험 설계 초안
         ▼
╔═════════════════════════╗
║   Phase 2: 피드백 루프   ║
║   연구자 ↔ research-    ║
║   planner               ║
║   (실험 계획 정교화)      ║  ◀─── 최대 5회 피드백
╚════════╤════════════════╝
         │ 확정된 실험 계획
         ▼
╔═════════════════════════╗
║   Phase 3: 실행 계획    ║
║   research-planner      ║
║   (구체적 실행 단계 수립)  ║
╚════════╤════════════════╝
         │ 승인
         ▼
╔═════════════════════════╗
║   Phase 4: 실험 실행    ║
║   research-executor     ║
║   ├ data-engineer       ║
║   └ 모델 실행           ║
║   (Baseline + Experiment ║
║    + Reference 실행)     ║
╚════════╤════════════════╝
         │ 실험 결과
         ▼
╔═════════════════════════╗
║   Phase 5: 결과 해석    ║
║   research-planner      ║
║   + paper-writer        ║
║   (비교 분석, 통계, 해석) ║
╚════════╤════════════════╝
         │ 결과 보고
         ▼
╔═════════════════════════╗
║   Phase 6: 후속         ║
║   - 추가 실험?          ║  ──▶ Phase 2로 복귀
║   - 논문 초안?          ║  ──▶ 경로 3으로 이동
║   - 종료?              ║
╚═════════════════════════╝
```

## 경로 3: 논문 초안 작성

실험 결과를 바탕으로 학술 논문 구조의 초안을 작성한다.

### 에이전트 구성표

| 에이전트 | 역할 |
|---|---|
| **paper-writer** | 논문 초안 작성 |
| **reviewer** | 논문 심사 (옵션) |

### 워크플로우

```
[연구자: "논문 초안 써줘"]
    │
    ▼
╔═════════════════════════╗
║   paper-writer          ║
║   (실험 결과 기반        ║
║    논문 초안 작성)       ║
╚════════╤════════════════╝
         │ 초안 완성
         ▼
╔═════════════════════════╗
║   reviewer (옵션)       ║
║   (논문 심사)           ║
╚════════╤════════════════╝
         │ 심사 결과
         ▼
╔═════════════════════════╗
║   수정/확정             ║
║   - 수정 필요 → paper-  ║
║     writer 재작성       ║
║   - 승인 → 최종 저장    ║
╚═════════════════════════╝
```

### 논문 초안 원칙

1. **객관적 서술**: 결과를 과장하지 않는다.
2. **수치 기반**: 주장에는 반드시 수치적 근거를 포함한다.
3. **그래프 참조**: 본문에서 Figure/Table을 명시적으로 참조한다.
4. **한계 투명성**: 실험의 한계를 솔직히 기술한다.
5. **초안 명시**: 이것은 초안이며, 연구자의 수정이 필요함을 명시한다.

### 언어

- **기본**: 영어 (국제 학술지 대상)
- **한국어**: 사용자 요청 시 한국어로 작성
- 사용자가 명시하지 않으면 영어로 작성하되, 확인한다.

## 에러 핸들링

| Phase | 에러 | 대응 |
|---|---|---|
| 1 | 요청/아이디어 해석 불가 | 사용자에게 구체적 질문 |
| 1 | 실현 불가능 요청 | 불가 사유 + 대안 제시 |
| 2 | 피드백 루프 최대 횟수 초과 | 최종 확정 요청 |
| 3 | 필요 데이터 미가용 | 가용한 데이터로 수정된 실험 제안 |
| 4 | 단계별 실패 | research-executor의 복구 전략 따름 |
| 4 | 전체 실패 | 부분 결과 보존 + 실패 보고서 생성 |
| 5 | 결과가 모호 | 추가 분석 또는 실험 반복 제안 |
| 논문 | 보고서 생성 실패 | 원시 결과 JSON을 직접 전달 |
