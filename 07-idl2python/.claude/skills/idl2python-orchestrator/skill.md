# IDL2Python-Orchestrator — IDL→Python 변환 파이프라인 총괄

## 개요

사용자가 IDL .pro 파일을 제시하면, 분석→변환→테스트→검토의 전 과정을 자동으로 조율한다.
에이전트 실행 순서, 데이터 전달 경로, 병렬 처리, 품질 게이트(PASS/REVISE), 사용자 승인 시점을 관리한다.

---

## 필수 입력 확인 및 검증

파이프라인 시작 전, 아래 항목을 확보한다. 누락 시 되물어서 확보한다.

| # | 항목 | 변수 | 누락 시 질문 |
|---|---|---|---|
| 1 | **IDL 파일 위치** | `{idl경로}` | "변환할 .pro 파일의 경로 또는 URL을 알려주세요. (로컬 경로, Git URL, 웹 URL 모두 가능)" |
| 2 | **변환 목적** | `{목적}` | "이 코드를 Python으로 변환하는 목적이 무엇인가요? (예: 레거시 마이그레이션, 특정 프로젝트 통합 등)" |
| 3 | **변환 모드** | `{모드}` | 아래 모드 선택지를 제시 |
| 4 | **작업 경로** | `{작업경로}` | "중간 결과물을 저장할 작업 경로를 알려주세요." |

### 소스 자동 판별

사용자가 `{idl경로}`를 제시하면 소스 유형을 자동 판별하고, Phase 0(수집)이 필요한 경우 자동으로 삽입한다:

```
{idl경로} 판별:
  │
  ├─ 로컬 파일/디렉토리  → Phase 0 생략, inbox/로 직접 복사
  ├─ github.com 등 Git   → Phase 0: git clone → .pro 추출
  └─ HTTP/FTP URL        → Phase 0: 디렉토리 파싱 → .pro 다운로드
                            (web-source-collector 스킬 사용)
```

### 변환 모드 선택지

```
변환 모드를 선택해주세요:

[1] 단일 변환 (Single)
    - 지정 .pro 파일 1개를 변환
    - 적합: 특정 파일 하나만 빠르게 변환할 때

[2] 배치 변환 (Batch)
    - 디렉토리/저장소/URL 내 모든 .pro 파일을 변환
    - 적합: IDL 프로젝트 전체를 마이그레이션할 때
    - 독립 파일은 병렬 처리

[3] 선택적 변환 (Selective)
    - 사용자가 지정한 파일 목록만 변환
    - 적합: 일부 파일만 선택적으로 변환할 때
```

## 모드별 에이전트 프리셋

### 단일 변환 (Single)

| Phase | 에이전트 | 프리셋 |
|---|---|---|
| 1 | idl-analyzer | 단일 파일 분석. 외부 의존성만 체크. |
| 2 | python-translator | 1개 파일 변환. |
| 3 | test-engineer | 단위 테스트 + 구문 검증. 합성 데이터 우선. |
| 4 | conversion-reviewer | 1개 파일 검토. |
| 5 | test-engineer | End-to-End 시연. 합성 데이터로 실행 + 물리 검증. |

### 배치 변환 (Batch)

| Phase | 에이전트 | 프리셋 |
|---|---|---|
| 1 | idl-analyzer | **핵심.** 전체 파일 스캔. 의존성 그래프 + 병렬 그룹 결정. |
| 2 | python-translator | 의존성 그래프 순서로 변환. 독립 파일은 병렬. |
| 3 | test-engineer | 모듈별 테스트 작성. 독립 모듈은 병렬 실행. |
| 4 | conversion-reviewer | 전체 통합 검토. |
| 5 | test-engineer | End-to-End 시연. 합성 데이터로 전체 파이프라인 연쇄 실행. |

### 선택적 변환 (Selective)

| Phase | 에이전트 | 프리셋 |
|---|---|---|
| 1 | idl-analyzer | 선택 파일만 분석. 미선택 파일의 의존성도 체크. |
| 2 | python-translator | 선택 파일만 변환. 외부 의존성은 stub 생성. |
| 3 | test-engineer | 선택 파일만 테스트. |
| 4 | conversion-reviewer | 선택 파일만 검토. |
| 5 | test-engineer | End-to-End 시연. 변환 범위 내에서 실행 가능한 시나리오 수행. |

## 작업 경로 설정

하네스 디렉토리의 `_workspace/`는 빈 스캐폴드(디렉토리 구조 템플릿)이다.
실행 결과물은 사용자가 지정한 외부 경로에 저장한다.

1. 사용자가 쿼리에 출력 경로를 명시한 경우 → 해당 경로를 `{작업경로}`로 사용
2. 경로가 확정되면 하위 디렉토리 생성:
   ```
   {작업경로}/
   ├── inbox/          # 원본 .pro 파일 복사본
   ├── analysis/       # IDL 분석 결과
   ├── converted/      # 변환된 Python 코드
   ├── tests/          # 테스트 코드
   ├── data/           # 테스트 데이터
   ├── reports/        # 검토 보고서
   └── logs/           # 변환 노트, 시스템 로그
   ```
3. 이후 모든 `_workspace/` 참조는 `{작업경로}`로 치환

### Phase 5: End-to-End 시연 (신규)

pytest PASS 후, 합성 데이터로 전체 파이프라인을 실제 실행하여 과학적 타당성을 검증한다.

| 단계 | 내용 |
|---|---|
| 5a | 합성 3D 자기장 생성 (testfield 또는 직접 구성) |
| 5b | 변환된 각 모듈을 연쇄 실행 |
| 5c | 결과를 알려진 해석적 답과 비교 |
| 5d | 에러 발생 시 → python-translator에 수정 요청 → 재테스트 → 재시연 |
| 5e | 시연 결과를 `reports/01_demo_report.md`에 기록 |

**시연에서 발견되는 전형적 버그:**
- 보간 함수의 경계 체크 누락 (IndexError)
- Newton-Raphson 집계 위치가 격자 밖 (median out of bounds)
- 유한 차분 스텐실이 경계를 넘음 (matrix_interp)
- IDL 원본 변수명 오류 전파 (NameError)

## 병렬 처리 관리

### 병렬 변환 (Phase 2)

idl-analyzer가 생성한 의존성 그래프를 기반으로 병렬 그룹을 결정:

```
analysis/01_dependency_graph.md 참조

그룹 A (독립): [utils.pro, constants.pro]     → 병렬 변환
그룹 B (A 의존): [data_loader.pro, plotter.pro] → A 완료 후 병렬
그룹 C (B 의존): [main_analysis.pro]            → B 완료 후 순차
```

**병렬 실행 방법:**
- Agent 도구를 사용하여 독립 파일들의 변환을 동시에 실행
- 각 Agent에게 단일 파일 변환을 위임하고, 모두 완료 후 다음 그룹 진행

### 병렬 테스트 (Phase 3)

독립 모듈의 테스트를 동시에 실행:
- `pytest -n auto` (pytest-xdist) 또는
- `concurrent.futures.ProcessPoolExecutor`

## 사용자 승인 시점

| 시점 | 내용 | 형식 |
|---|---|---|
| Phase 1 완료 후 | 변환 계획 + 난이도 + 예상 이슈 | `analysis/03_conversion_plan.md` 요약 제시 |
| Phase 4 PASS 후 | 최종 변환 결과 + 테스트 결과 | `reports/00_review_report.md` 요약 제시 |

**승인 질문 형식:**

```
변환 계획이 나왔습니다:

총 N개 파일 분석 완료:
- 변환 대상: M개
- 제외: K개 (사유: GUI 위젯 전용 등)
- 난이도: 상 X개, 중 Y개, 하 Z개

주요 이슈:
1. column-major 배열 변환 필요 (P건)
2. SSW 의존성 매핑 필요 (Q건)

병렬 처리 가능: A그룹 N개, B그룹 M개

이 계획대로 진행할까요? 수정이 필요하면 알려주세요.
```

## 루프백 조건과 최대 횟수

| 루프백 유형 | 트리거 | 최대 횟수 | 초과 시 |
|---|---|---|---|

| **변환 계획 수정** | 사용자가 계획 수정 요청 | 1회 | 수정 반영 후 진행 |

## 에러 핸들링 테이블

| Phase | 에러 유형 | 심각도 | 대응 |
|---|---|---|---|
| 준비 | .pro 파일 없음 | 높음 | 경로 재확인 요청 |
| 준비 | .pro 파일 인코딩 오류 | 중간 | 인코딩 자동 감지, 실패 시 사용자 확인 |
| 1 | 파일 수 과다 (>50개) | 중간 | 우선순위 분류 후 범위 확인 |
| 1 | IDL 문법 오류 (원본) | 중간 | 가능한 범위에서 분석, 오류 위치 보고 |
| 2 | 매핑 불가 IDL 구문 | 중간 | TODO 주석 + 사용자 알림 |
| 2 | SSW 루틴 미식별 | 중간 | idl-python-mapping 참조, 없으면 stub 생성 |
| 3 | 테스트 데이터 다운로드 실패 | 중간 | 합성 데이터로 대체 |
| 3 | pytest 미설치 | 낮음 | 자동 설치 안내 |
| 4 | REVISE 2회 초과 | 높음 | 사용자 에스컬레이션 |

## conversion-note.md 기록 규칙

모든 에이전트는 `{작업경로}/logs/conversion-note.md`에 판단 과정을 누적 기록한다.

```markdown
# Conversion Note

## 변환 개요
- **IDL 파일**: {idl경로}
- **목적**: {목적}
- **모드**: {모드}
- **시작일**: {timestamp}

---

## [Phase 1: IDL 분석] {timestamp}
### idl-analyzer
- 분석 전략: ...
- 의존성 추론 과정: ...
- 병렬 그룹 결정 근거: ...

## [Phase 2: Python 변환] {timestamp}
### python-translator
- 매핑 선택 근거: ...
- 인덱싱 변환 상세: ...

## [Phase 3: 테스트] {timestamp}
### test-engineer
- 테스트 전략: ...
- 데이터 선택 근거: ...

## [Phase 4: 품질 검토] {timestamp}
### conversion-reviewer
- 판정 근거: ...
- 인덱싱 검증 상세: ...
```

**규칙:**
- 각 에이전트는 자기 Phase 섹션에만 추가한다
- 기존 내용을 수정하지 않고 누적만 한다
- 판단의 **이유**를 반드시 기록한다

## 최종 산출물 안내

검증 통과(PASS)한 변환 결과를 사용자에게 안내한다:

```
변환이 완료되었습니다!

변환 결과:
1. {module_name}.py — {설명} (원본: {module_name}.pro)
2. {module_name}.py — {설명} (원본: {module_name}.pro)

테스트 결과: 전체 K개 중 P개 PASS

파일 위치:
- 변환 코드: {작업경로}/converted/
- 테스트 코드: {작업경로}/tests/
- 의존성: {작업경로}/converted/requirements.txt
- 검토 보고서: {작업경로}/reports/00_review_report.md

사용법:
  pip install -r {작업경로}/converted/requirements.txt
  python -c "from converted.{module_name} import ..."
  pytest {작업경로}/tests/ -v
```
