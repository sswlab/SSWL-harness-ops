# Code-Analysis-Protocol — 코드 분석 절차 및 인벤토리 표준

## 개요

연구실 코드는 개인 취향으로 작성되어 문서화가 부족하고, 하드코딩이 많으며,
하나의 파일이 여러 목적을 갖는 경우가 흔하다. 이 스킬은 그런 코드를
체계적으로 분석하는 절차를 정의한다.

## 분석 3단계 절차

### 1단계: 빠른 스캔 (Quick Scan)

모든 파일에 대해 1차 분류를 수행한다. 파일을 열지 않고도 파악 가능한 정보를 추출한다.

```
수집 항목:
- 파일명, 확장자, 크기, 최종 수정일
- 1~10줄 (import 문, shebang, docstring)
- 파일명에서 추론 가능한 목적
```

**1단계 결과로 우선순위를 분류한다:**
- **A등급**: import 문에 도메인 라이브러리(sunpy, astropy 등) 포함 → 상세 분석
- **B등급**: 일반 라이브러리만 사용 → 상세 분석 (우선순위 낮음)
- **C등급**: 빈 파일, 바이너리, 데이터 파일 → 스킵

### 2단계: 상세 분석 (Deep Analysis)

A/B등급 파일을 대상으로 전체 코드를 읽고 분석한다.

**함수/클래스 추출 기준:**
- 모든 `def`, `class` 정의
- `if __name__ == "__main__"` 블록
- 전역 스코프의 주요 실행 코드

**각 함수/클래스에 대해:**

| 항목 | 추출 방법 |
|---|---|
| 목적 | docstring → 함수명 → 코드 흐름 순으로 추론 |
| 입력 | 파라미터 타입힌트 → 내부 사용 패턴 → 호출부 역추적 |
| 출력 | return문 → 파일 쓰기 → plt.show() → print() |
| 의존성 | import문 + 함수 내 사용 |
| 부작용 | 파일 생성, 전역 변수 수정, API 호출 |

### 3단계: 관계 분석 (Relationship Analysis)

파일 간, 함수 간 관계를 파악한다.

**의존 관계 유형:**
- **직접 import**: `from file_a import func` → 강한 의존
- **파일 전달**: file_a가 생성한 파일을 file_b가 읽음 → 간접 의존
- **복제 코드**: 동일/유사 로직이 여러 파일에 존재 → 중복 관계

## 재사용 가치 평가 기준

| 등급 | 기준 | 예시 |
|---|---|---|
| **높음** | 범용적 기능, 여러 프로젝트에서 사용 가능 | 데이터 로더, 플롯 유틸리티, 전처리 파이프라인 |
| **중간** | 특정 도메인에서 재사용 가능 | STIX 전용 분석, 특정 모델 래퍼 |
| **낮음** | 일회성 또는 특정 실험에 종속 | 특정 날짜 데이터 분석, 임시 테스트 |

**낮음 판정 근거 (하나라도 해당하면):**
- 하드코딩된 날짜/경로가 핵심 로직에 포함
- 특정 실험 결과에 대한 일회성 후처리
- `test_`, `tmp_`, `old_`, `backup_` 접두사

## 불확실성 처리 규칙

코드의 목적이 불분명할 때:

1. `[불확실]` 태그를 명시적으로 부여한다
2. 가능한 해석을 최대 3가지 제시한다
3. 판단 근거를 기록한다 (어떤 단서에서 어떤 추론을 했는지)
4. 사용자에게 확인을 요청하는 질문을 구체적으로 작성한다

```markdown
### file_x.py [불확실]
- **추정 목적**: 
  1. GOES 라이트커브 분류기 (근거: goes_class 변수)
  2. 플레어 강도 변환기 (근거: flux_to_class 함수)
  3. 데이터 라벨링 도구 (근거: label 딕셔너리)
- **확인 질문**: "file_x.py는 GOES 플레어 등급을 분류하는 코드인가요, 아니면 라벨링을 위한 유틸리티인가요?"
```
